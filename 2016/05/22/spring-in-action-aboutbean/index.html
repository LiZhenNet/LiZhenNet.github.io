Spring In Action 笔记 --关于Bean

Spring

Bean 容器

创建Bean 分发Bean

BeanFactory

常用实现：XmlBeanFactory

ApplicationContext

比BeanFactory多的功能：

  1. 文本信息的解析，国际化的支持
  2. 载入文件资源的通用方法
  3. 向注册为监听器的Bean发送事件

常用实现：

  1. ClassPathXmlApplicationContext 从类路径查找xml

  2. FileSystemXmlApplicationContext 从文件系统查找xml

  3. XmlWebAppliactionContext 从web系统获取xml

BeanFactory 在调用getBean时候创建Bean ，ApplicationContext会预加载所有单实例Bean

Bean的生命

Bean的创建过程：

  1. 容器寻找Bean的定义信息并将其实例化
  2. 使用依赖注入，Spring按照Bean定义信息配置Bean的所有属性
  3. 如果Bean实现了 BeanNameAware接口，工厂调用Bean的setBeanName()方法传递Bean的ID
  4. 如果Bean实现了 BeanFactoryAware接口，工厂调用setBeanFactory方法传入工厂本身
  5. 如果有BeanPostProcessor和Bean关联，那么他们的postProcessBeforeInitialzation方法将被调用
  6. 如果Bean 指定了init-method方法， 它将被调用
  7. 如果有BeanPostProcessor和Bean关联，那么他们的postProcessAfterInitialzation方法将被调用

Bean的删除过程：

  1. 如果Bean实现了DisposableBean接口，destory（）方法将被调用
  2. 如果指定了销毁方法就调用这个方法

Appliication 创建Bean 的时候如果实现了ApplicationContextAware接口，setApplicationContext将会在setBeanFactory后调用

XML一些配置

singleton 默认为true 单例模式， 为false时为原型模式 即每次返回一个新实例
init-method 实例化后调用
destroy-method 从容器删除时调用

Bean注入

Set方法注入

构造函数注入

为了解决Bean注入参数的不确定性，可以使用index来指定参数顺序，type来指定参数类型

自动注入

Bean添加属性autowire=”byName|byType|constructor|autodetect”

  1. byName 试图在容器中寻找自动装的属性名相同的Bean
  2. byType 试图在容器中寻找自动装的属性类型相同的Bean
  3. constructor 试图在容器中查找和需要自动装配的Bean的构造函数一致的一个或多个Bean
  4. autodetect 首先尝试 constructor ，然后使用byType

Bean Type转换

在前面的例子中也使用了 type属性来告诉Spring传入参数的类型是什么。但是我们传入的都是String类型，他是如何转换的呢。
是通过继承 java.Beans.PropertyEditorSupport 重写调用setAsText来实现的。

事件

监听事件

在生命周期中ApplicationContext 会发生很多事件这些事件都是org.springframework.context.ApplicationEvent的子类
系统事件有：

  1. ContextRefreshedEvent 在应用上下文初始化或者刷新的时候发布
  2. ContextClosedEvent 在应用上下文关闭的时候发布
  3. RequestHandledEvent 在Web应用中，当一个请求呗处理后发布
    当我们需要监听这些时间时，我们需要实现org.springframework,context.ApplicationListener接口
