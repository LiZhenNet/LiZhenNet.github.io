理解Spring--Ioc与DI

Java
  • Di
  • Ioc
  • Java
  • Spring

本来想每月最少两篇博客，结果这个月比较忙（借口），只能月末来篇凑数，下个月再补上~~

在理解Ioc和Di之前，我们先了解什么是DIP（依赖倒置原则）

依赖倒置原则 （Dependence Inversion Principle）

上层模块不应该依赖于下层模块，它们共同依赖于一个抽象。
抽象不能依赖于具象，具象依赖于抽象。

如何理解依赖倒置原则呢 ，就是上层模块就是使用者，下层模块是被使用者，上层模块依赖下层模块，
下层模块的变动势必会影响上层模块，所以上层模块不应该依赖于下层模块，它们共同依赖于一个抽象，
这里所说的抽象也就是接口。

控制反转（Inversion of Control）

它把传统上由程序代码直接操控的对象的调用权交给容器，通过容器来实现对象组件的装配和管理。
所谓的“控制反转”概念就是对组件对象控制权的转移，从程序代码本身转移到了外部容器。

为了理解这一概念我们看下图:

我之前出门是自己拿衣服然后出门（使用者直接调用被使用者），使用Ioc之后，多了 “女朋友(容器)” 这一角色，
她根据我们给她的信号来把衣服拿给我们（也就是调用被使用者这件事交给容器来处理）。

依赖注入（Dependency Injection）

许多资料把IoC和Di 当做同等概念， 实际上Di 只是Ioc常用的实现方式，Ioc另一个实现方式就是依赖查找（Dependency Lookup）。

依赖注入，就是由IoC容器在运行期间，动态地将某种依赖关系注入到对象之中。如何注入到对象中呢？
有如下三种方式：

  1. 接口注入
  2. 构造方法注入
  3. setter方法注入

下面简述一下这三种方式：

构造方法注入

这种注入方式的优点就是，对象在构造完成之后，即已进入就绪状态，可以马上使用。
缺点就是，当依赖对象比较多的时候，构造方法的参数列表会比较长。而通过反射构造对象的时候，对相同类型的参数的处理会比较困难，维护和使用上也比较麻烦。
而且在Java中，构造方法无法被继承，无法设置默认值。对于非必须的依赖处理，可能需要引入多个构造方法，而参数数量的变动可能造成维护上的不便。

setter方法注入

因为方法可以命名，所以setter方法注入在描述性上要比构造方法注入好一些。
另外，setter方法可以被继承，允许设置默认值，而且有良好的IDE支持。
缺点当然就是对象无法在构造完成后马上进入就绪状态。

接口注入

从注入方式的使用上来说，接口注入是现在不甚提倡的一种方式，基本处于“退役状态”。
因为它强制被注入对象实现不必要的接口，带有侵入性。而构造方法注入和setter方法注入则不需要如此。
